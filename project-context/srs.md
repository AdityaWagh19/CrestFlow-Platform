## 1. Introduction

### 1.1 Purpose

This document defines the functional and non-functional requirements for CrestFlow — an AI-native financial intelligence and portfolio orchestration platform built on Algorand.

The system aggregates on-chain portfolio data, analyzes risk, discovers yield opportunities, generates portfolio strategies, builds user investor profiles, and executes user-approved actions through integrated Algorand protocols.

This SRS is the authoritative technical reference for all subsystems. All requirements are grounded in the finalized architecture defined in `system-architecture.png`.

---

### 1.2 Scope

The system shall provide:

- **User Onboarding & Identity** — Auth, embedded wallet creation, KYC, and DID/VC issuance
- **Portfolio Intelligence** — On-chain position discovery and normalized portfolio analytics
- **Risk Intelligence** — Portfolio risk quantification, stress testing, and alerts
- **Strategy Intelligence** — Portfolio optimization and goal-based strategy generation
- **Yield Intelligence** — Opportunity discovery, ranking, and sustainability analysis
- **User Intelligence** — Dynamic investor profiling and behavioral personalization
- **Autonomous Execution** — Intent routing, transaction generation, and approved execution
- **Orchestration Layer** — Policy enforcement, planning, and cross-protocol routing
- **AI Copilot** — Natural language interface over all engines
- **Programmatic Access** — REST APIs with x402 monetization and MCP interfaces

The system shall support both direct user interaction (via the AI Copilot UI) and programmatic access (via Copilot API endpoints and MCP tools).

---

### 1.3 Definitions

| Term | Definition |
|---|---|
| **Portfolio** | The complete collection of a user's on-chain assets and protocol positions |
| **Position** | User participation in a protocol (lending, borrowing, LP, staking) |
| **Exposure** | Economic ownership after decomposing wrapped assets, LP tokens, and vaults into underlying assets |
| **Portfolio Snapshot** | Point-in-time, normalized representation of the complete user portfolio state |
| **Health Score** | A composite 0–100 metric representing overall portfolio quality |
| **Risk Score** | A composite 0–100 metric representing overall portfolio risk |
| **Investor Persona** | A behavioral classification of a user's risk tolerance and investment style |
| **POA** | Plan of Action — the ordered execution plan generated by the Orchestrator/Planner |
| **VC** | Verifiable Credential — a cryptographically signed KYC credential |
| **DID** | Decentralized Identifier — user identity anchor on-chain |
| **x402** | HTTP 402-based micropayment protocol for per-call API monetization |
| **MCP** | Model Context Protocol — standard for exposing tools to external AI agents |
| **HRP** | Hierarchical Risk Parity — portfolio optimization method |
| **HHI** | Herfindahl-Hirschman Index — concentration measurement method |
| **VaR** | Value at Risk — risk measurement metric |
| **CVaR** | Conditional Value at Risk — tail risk measurement |
| **TVL** | Total Value Locked — measure of protocol liquidity |
| **ASA** | Algorand Standard Asset |
| **LP** | Liquidity Provider position |
| **AML** | Anti-Money Laundering |

---

### 1.4 Constraints

- All financial execution requires **explicit user authorization**. No autonomous execution without approval.
- The system is **non-custodial**. Private keys are never stored or accessed by CrestFlow.
- All AI recommendations must be **explainable** — rationale, confidence, assumptions, and expected outcomes must be surfaced.
- No engine may **directly depend** on another engine's internal implementation. Inter-engine communication occurs only through defined contracts.
- All execution actions must pass through the **Policy Engine** before routing.

## 2. System Overview

CrestFlow consists of the following major subsystems:

| # | Subsystem | Role |
|---|---|---|
| — | Onboarding & Identity | Auth, wallet creation, KYC, DID/VC |
| — | Financial Knowledge Layer | Shared data foundation for all engines |
| 1 | Portfolio Intelligence Engine | Normalize on-chain data into portfolio intelligence |
| 2 | Risk Intelligence Engine | Quantify and monitor portfolio risk |
| 3 | Strategy & Optimization Engine | Generate portfolio strategies and rebalancing plans |
| 4 | Yield & Opportunity Engine | Discover and rank yield opportunities |
| 5 | User Intelligence Engine | Build and evolve dynamic investor profiles |
| 6 | Autonomous Execution Engine | Generate and execute approved portfolio actions |
| — | Orchestration & Execution Layer | Policy, planning, coordination, and protocol routing |
| — | AI Copilot Layer | Natural language interface over all engines |
| — | Copilot API | Unified public API with x402 monetization |

**Architectural rules:**
- Each engine shall be independently deployable
- Engines shall communicate through versioned API contracts
- All engines are exposed via the Copilot API — the single public endpoint
- The Financial Knowledge Layer is a shared data foundation consumed by all engines

## 3. System Architecture

### 3.1 Full System Flow

```
User
 │
 ├── Auth (Google OAuth / Email+Password)
 │
 ▼
Onboarding & Identity
 ├── Turnkey Embedded Wallet
 ├── KYC (Veriff)
 └── DID + VC (GoPlausible)
 │
 ▼
Portfolio State Analysis
(Algorand Indexer → all positions discovered)
 │
 ▼
─────────────────────────────────────────────────────────────
CORE SERVICES  (Copilot API — routes to 6 engines via x402)
─────────────────────────────────────────────────────────────
 │
 ├── Financial Knowledge Layer (shared data foundation)
 │     └── Market data, protocol data, historical data,
 │         token data, risk data, research heuristics
 │
 ├── Engine 1: Portfolio Intelligence Engine
 ├── Engine 2: Risk Intelligence Engine
 ├── Engine 3: Strategy & Optimization Engine
 ├── Engine 4: Yield & Opportunity Engine
 ├── Engine 5: User Intelligence Engine
 └── Engine 6: Autonomous Execution Engine
 │
 ▼
Orchestration & Execution Layer
 ├── x402 Payment → GoPlausible Facilitator
 ├── Policy Engine (guardrails)
 ├── Orchestrator / Planner (POA)
 └── Execution Coordinator
           │
           ▼
      Haystack Router
           │
      ┌────┴─────────────────────┐
      │                          │
  Gora Oracle             Protocol Adapters
  (price feeds)      Folks Finance / Tinyman / Pact
                              │
                              ▼
                  Wallet Transactions on Algorand
                  (Fully auditable, instant finality)
```

### 3.2 Engine Communication Contract

- Engines do **not** call each other's internal methods
- Engine 1 (Portfolio Intelligence) produces the **canonical financial state** consumed by all other engines
- Engine 5 (User Intelligence) produces the **investor profile** consumed by Engines 2, 3, 4
- All inter-engine data flows through defined output schemas

## 4. Onboarding & Identity Subsystem

### 4.1 Authentication

**REQ-AUTH-01:** The system shall support authentication via Google OAuth.  
**REQ-AUTH-02:** The system shall support authentication via email and password.  
**REQ-AUTH-03:** The system shall create a platform user record upon first successful authentication.

---

### 4.2 Embedded Wallet

> ★ MVP Priority — from `system-architecture.png`

**REQ-WALLET-01:** The system shall create an embedded Algorand wallet for each user via Turnkey upon successful account creation.  
**REQ-WALLET-02:** The system shall store the wallet address linked to the user account.  
**REQ-WALLET-03:** The system shall never store or access user private keys; Turnkey's non-custodial architecture shall be enforced.  
**REQ-WALLET-04:** The system shall use the Turnkey embedded wallet as the sole transaction-signing wallet. External wallet connections are not supported — they cannot autonomously pay x402 fees.

---

### 4.3 KYC Infrastructure

> Optional for MVP. Required production gate before execution features are enabled.

**REQ-KYC-01:** The system shall integrate with Veriff as the KYC provider.  
**REQ-KYC-02:** The KYC flow shall include: document verification (Aadhaar/PAN or equivalent), liveness check, and AML screening.  
**REQ-KYC-03:** KYC verification shall be a configurable gate — toggleable between optional (MVP) and required (production) via system configuration.  
**REQ-KYC-04:** The system shall record KYC status per user and expose it to the Policy Engine for execution gating.

---

### 4.4 Decentralized Identity & Verifiable Credentials

**REQ-DID-01:** Upon successful KYC, the system shall generate a Decentralized Identifier (DID) for the user via GoPlausible.  
**REQ-DID-02:** The system shall issue a KYC Verifiable Credential (VC) linked to the user's DID.  
**REQ-DID-03:** The user's DID shall be connected to their Algorand wallet following the chain: `wallet → did → vc`.  
**REQ-DID-04:** The VC shall be usable by the Policy Engine as an authorization signal.

---

### 4.5 On-Ramp & Funding

**REQ-ONRAMP-01:** The system shall support UPI-based fiat on-ramp for wallet funding.

---

### 4.6 Post-Onboarding Portfolio Scan

**REQ-SCAN-01:** Upon completing onboarding, the system shall automatically trigger a full portfolio state analysis.  
**REQ-SCAN-02:** The portfolio scan shall discover all native asset holdings, protocol positions, and LP positions for the connected wallet(s).  
**REQ-SCAN-03:** The scan output shall serve as the base data feed for all six engines.

---

## 5. Financial Knowledge Layer

The Financial Knowledge Layer is a shared data foundation. It is not a user-facing engine. All six engines consume from it.

**REQ-FKL-01:** The layer shall aggregate live market data from CoinGecko.  
**REQ-FKL-02:** The layer shall provide verified price feeds via Gora Oracle.  
**REQ-FKL-03:** The layer shall aggregate protocol data (TVL, APYs, liquidity) from Folks Finance, Tinyman, and Pact APIs.  
**REQ-FKL-04:** The layer shall provide on-chain data via the Algorand Indexer.  
**REQ-FKL-05:** The layer shall maintain historical data for performance and trend analysis.  
**REQ-FKL-06:** The layer shall maintain token metadata and risk data for asset classification.  
**REQ-FKL-07:** Data from this layer shall be versioned and timestamped to ensure auditability.

## 6. Engine 1 — Portfolio Intelligence Engine

**Purpose:** Convert raw on-chain data into normalized portfolio intelligence. Produces the canonical financial state consumed by all downstream engines.

> Sample APIs: `GET /analyze`, `GET /health`, `GET /performance`  
> Sample SOTA: Factor Analysis, Performance Attribution Models

---

### 6.1 Asset Discovery

**REQ-PIE-01:** The system shall discover all native ALGO and ASA holdings for a given wallet address using the Algorand Indexer.  
**REQ-PIE-02:** The system shall discover all active positions in Folks Finance (lending, borrowing).  
**REQ-PIE-03:** The system shall discover all active LP positions in Tinyman and Pact.  
**REQ-PIE-04:** The system shall discover staking positions where applicable.  
**REQ-PIE-05:** The system shall normalize all discovered positions into a unified asset record schema.

---

### 6.2 Portfolio Aggregation

**REQ-PIE-06:** The system shall calculate the total portfolio value in USD and ALGO.  
**REQ-PIE-07:** The system shall aggregate all positions across wallets and protocols into a single portfolio snapshot.  
**REQ-PIE-08:** The system shall timestamp each portfolio snapshot for historical tracking.

**Output:** Portfolio Snapshot (canonical financial state for downstream engines)

---

### 6.3 Allocation Analysis

**REQ-PIE-09:** The system shall calculate asset allocation as a percentage of total portfolio value.  
**REQ-PIE-10:** The system shall calculate category allocation (stablecoin, volatile, LP, staking).  
**REQ-PIE-11:** The system shall calculate protocol allocation (% in Folks Finance vs. Tinyman vs. Pact vs. native).

**Output:** Allocation Breakdown

---

### 6.4 Exposure Analysis

**REQ-PIE-12:** The system shall calculate direct asset exposure (tokens held natively).  
**REQ-PIE-13:** The system shall calculate indirect exposure by decomposing LP tokens into underlying assets.  
**REQ-PIE-14:** The system shall calculate indirect exposure from vault or yield-bearing positions.  
**REQ-PIE-15:** The system shall surface the true underlying asset exposure as a percentage of total portfolio.

**Output:** Exposure Report (e.g., "72% of your portfolio is indirectly exposed to ALGO")

---

### 6.5 Performance Analysis

**REQ-PIE-16:** The system shall track portfolio value over time using historical snapshots.  
**REQ-PIE-17:** The system shall calculate absolute and percentage returns over standard periods (7D, 30D, 90D, All-Time).  
**REQ-PIE-18:** The system shall attribute performance by asset and by protocol (performance attribution).

**Output:** Performance Metrics

---

### 6.6 PnL Calculation

**REQ-PIE-19:** The system shall calculate realized PnL from closed positions and completed swaps.  
**REQ-PIE-20:** The system shall calculate unrealized PnL from open positions at current market prices.  
**REQ-PIE-21:** The system shall calculate yield earned from lending, LP rewards, and staking.  
**REQ-PIE-22:** The system shall calculate fees paid across protocol interactions.

**Output:** PnL Report

---

### 6.7 Cost Basis Tracking

**REQ-PIE-23:** The system shall support FIFO (First In, First Out) cost basis method.  
**REQ-PIE-24:** The system shall support LIFO (Last In, First Out) cost basis method.  
**REQ-PIE-25:** The system shall support Average Cost basis method.  
**REQ-PIE-26:** The system shall allow the user to select their preferred cost basis method.

**Output:** Cost Basis Report

---

### 6.8 Portfolio Health Score

**REQ-PIE-27:** The system shall generate a Portfolio Health Score in the range 0–100.  
**REQ-PIE-28:** The Health Score shall be a composite of: diversification, concentration risk, liquidity adequacy, risk-adjusted yield, yield sustainability, and protocol quality.  
**REQ-PIE-29:** The system shall provide a per-component breakdown of the Health Score.  
**REQ-PIE-30:** The system shall identify and surface the top strengths and weaknesses driving the score.  
**REQ-PIE-31:** The system shall generate at least one actionable improvement suggestion per weakness identified.

**Output:** Health Score (0–100) + breakdown + strengths + weaknesses + actions

---

## 7. Engine 2 — Risk Intelligence Engine

**Purpose:** Measure, monitor, and explain all material portfolio risks.

> Sample APIs: `POST /risk/stress-test`, `GET /risk/liquidation`, `GET /risk/score`  
> Sample SOTA: Monte Carlo simulation, Bayesian models, Regime models

---

### 7.1 Risk Scoring

**REQ-RIE-01:** The system shall generate an overall portfolio Risk Score in the range 0–100 (0 = lowest risk).  
**REQ-RIE-02:** The system shall generate an asset-level risk score for each holding.  
**REQ-RIE-03:** The system shall generate a protocol-level risk score for each integrated protocol.  
**REQ-RIE-04:** Each risk score shall include an explanation of the primary contributing factors.

**Output:** Risk Score (overall, asset-level, protocol-level) + explanations

---

### 7.2 Concentration Analysis

**REQ-RIE-05:** The system shall calculate asset concentration using Herfindahl-Hirschman Index (HHI) or equivalent portfolio weight analysis.  
**REQ-RIE-06:** The system shall calculate protocol concentration (% of portfolio in each protocol).  
**REQ-RIE-07:** The system shall flag concentration above configurable thresholds as a risk alert.

**Output:** Concentration Report + risk alerts

---

### 7.3 Liquidity Analysis

**REQ-RIE-08:** The system shall evaluate market liquidity for each held asset.  
**REQ-RIE-09:** The system shall evaluate pool liquidity for LP positions (depth and slippage estimates).  
**REQ-RIE-10:** The system shall assess exit feasibility — the ability to exit positions within a reasonable time and cost.  
**REQ-RIE-11:** The system shall generate a Liquidity Score per position and at the portfolio level.

**Output:** Liquidity Score + exit feasibility assessment

---

### 7.4 Drawdown Analysis

**REQ-RIE-12:** The system shall calculate the maximum historical drawdown for the portfolio.  
**REQ-RIE-13:** The system shall calculate the current drawdown from the portfolio's all-time high.  
**REQ-RIE-14:** The system shall track historical drawdown episodes.

**Output:** Drawdown Metrics

---

### 7.5 Stress Testing

**REQ-RIE-15:** The system shall support configurable stress test scenarios.  
**REQ-RIE-16:** The system shall include at minimum the following built-in scenarios:

| Scenario | Description |
|---|---|
| Market Crash | Broad market decline (e.g., ALGO −40%, majors −60%) |
| Stablecoin Depeg | Stablecoin loses peg (e.g., USDC falls to $0.85) |
| Liquidity Shock | Protocol TVL drops sharply; exit liquidity collapses |
| Protocol Failure | A specific integrated protocol becomes inoperable |

**REQ-RIE-17:** Each scenario result shall include estimated portfolio value impact, positions most affected, and suggested mitigations.

**Output:** Scenario Impact Report per scenario

---

### 7.6 Liquidation Monitoring

**REQ-RIE-18:** For Folks Finance borrowing positions, the system shall calculate proximity to liquidation threshold.  
**REQ-RIE-19:** The system shall generate a liquidation alert when a position is within a configurable threshold of the liquidation price.  
**REQ-RIE-20:** The system shall recommend specific actions to reduce liquidation risk when triggered.

**Output:** Liquidation proximity indicator + alerts

---

## 8. Engine 3 — Strategy & Optimization Engine

**Purpose:** Generate explainable, data-driven portfolio strategies.

> Sample APIs: `GET /strategy/allocation`, `POST /strategy/rebalance`, `GET /strategy/recommendations`  
> Sample SOTA: Black-Litterman, Hierarchical Risk Parity (HRP), Risk Parity

---

### 8.1 Portfolio Optimization

**REQ-SOE-01:** The system shall generate asset rebalancing plans with specific, actionable steps.  
**REQ-SOE-02:** The system shall generate allocation improvement recommendations based on the user's current portfolio and risk profile.  
**REQ-SOE-03:** The system shall generate yield optimization plans that improve risk-adjusted return without increasing overall risk beyond the user's profile.  
**REQ-SOE-04:** Each optimization output shall quantify the expected improvement (e.g., "+1.8% projected yield", "−12 risk score points").

**Output:** Strategy Recommendations + expected outcomes

---

### 8.2 Goal-Based Planning

**REQ-SOE-05:** The system shall support the following portfolio goals:

| Goal | Description |
|---|---|
| Capital Preservation | Minimize downside; prioritize stablecoins and low-risk yield |
| Income Generation | Maximize sustainable yield; balance risk |
| Growth | Maximize long-term upside; accept higher volatility |
| Balanced | Optimize risk-adjusted returns across all dimensions |

**REQ-SOE-06:** The system shall generate strategies aligned to the user's selected goal, informed by the investor profile from Engine 5.

**Output:** Goal-Aligned Strategies

---

### 8.3 Recommendation Explainability

Every strategy recommendation generated by this engine shall include:

**REQ-SOE-07:** The reason the recommendation exists (data-backed rationale)  
**REQ-SOE-08:** The expected outcome (projected return, risk change, or capital efficiency improvement)  
**REQ-SOE-09:** The risks associated with taking the recommended action  
**REQ-SOE-10:** A confidence score indicating model certainty  
**REQ-SOE-11:** The assumptions made in generating the recommendation

---

## 9. Engine 4 — Yield & Opportunity Engine

**Purpose:** Discover, evaluate, and rank yield opportunities across Algorand protocols.

> Sample APIs: `GET /yield/opportunities`, `GET /yield/rankings`, `POST /yield/optimize`  
> Sample SOTA: Risk-Adjusted Yield Scoring

---

### 9.1 Opportunity Discovery

**REQ-YOE-01:** The system shall discover lending yield opportunities across Folks Finance.  
**REQ-YOE-02:** The system shall discover LP yield opportunities across Tinyman and Pact.  
**REQ-YOE-03:** The system shall discover staking opportunities where available on Algorand.  
**REQ-YOE-04:** The system shall detect idle capital (assets generating no yield) and flag it with suggestions.  
**REQ-YOE-05:** Opportunity discovery shall be filtered by the user's risk profile from Engine 5.

**Output:** Opportunity List (filtered and ranked)

---

### 9.2 Opportunity Ranking

**REQ-YOE-06:** The system shall rank all discovered opportunities across the following dimensions:

| Dimension | Description |
|---|---|
| Raw APY | Nominal annual percentage yield |
| Risk-Adjusted APY | APY discounted by protocol and market risk |
| Liquidity | Pool depth and exit liquidity |
| Sustainability Score | Estimated longevity of the yield source |

**REQ-YOE-07:** The system shall expose both a raw APY ranking and a risk-adjusted ranking.  
**REQ-YOE-08:** The default ranking surfaced to users shall be risk-adjusted, aligned to the user's risk profile.

**Output:** Ranked Opportunity List

---

### 9.3 Yield Sustainability Analysis

**REQ-YOE-09:** For each opportunity, the system shall estimate a sustainability score indicating yield longevity.  
**REQ-YOE-10:** The system shall identify yield sources with high reward dependency (yield driven primarily by token emissions that may decline).  
**REQ-YOE-11:** The system shall flag unsustainable yield sources with an appropriate warning.

**Output:** Sustainability Report per opportunity

---

## 10. Engine 5 — User Intelligence Engine

**Purpose:** Build, maintain, and continuously evolve a dynamic investor profile for each user. Personalizes the output of all other engines.

> Sample APIs: `GET /profile`, `GET /preferences`, `GET /persona`  
> Sample SOTA: Behavioral Finance models, Preference Learning

---

### 10.1 Investor Persona Classification

**REQ-UIE-01:** The system shall classify each user into an investor persona based on stated preferences and observed behavior.  
**REQ-UIE-02:** The system shall support at minimum the following persona categories:

| Persona | Description |
|---|---|
| Conservative | Risk-averse; prefers stablecoins and low-volatility yield |
| Balanced | Accepts moderate risk for better returns |
| Aggressive | Accepts high volatility for maximum upside |

**REQ-UIE-03:** The persona shall be displayable to the user with a clear explanation.

---

### 10.2 Dynamic Risk Profile

**REQ-UIE-04:** The system shall maintain a dynamic risk profile per user, updated continuously based on behavior.  
**REQ-UIE-05:** Behavioral signals that update the risk profile shall include: recommendations accepted/rejected, queries made, portfolio changes observed, and explicitly stated preferences.  
**REQ-UIE-06:** The risk profile shall be passed to Engines 2, 3, and 4 to personalize their outputs.

---

### 10.3 Goal Tracking

**REQ-UIE-07:** The system shall allow users to set explicit financial goals (yield target, growth target, capital preservation).  
**REQ-UIE-08:** The system shall track progress toward stated goals over time.  
**REQ-UIE-09:** Goal data shall be passed to Engine 3 (Strategy & Optimization) for strategy alignment.

---

### 10.4 Preference Learning

**REQ-UIE-10:** The system shall learn from user interactions — which recommendations are accepted, which are dismissed, and which queries are repeated.  
**REQ-UIE-11:** Learned preferences shall be used to improve the relevance and personalization of all engine outputs.

**Output:** Investor Persona + Dynamic Risk Profile + Goal Progress

---

## 11. Engine 6 — Autonomous Execution Engine

**Purpose:** Generate and execute approved portfolio actions through integrated Algorand protocols.

> Sample APIs: `POST /execute`, `POST /execution-plan`, `POST /autopilot`  
> Sample SOTA: Intent routing, Agentic execution patterns

> **Critical constraint:** This engine shall never execute any action without explicit user authorization routed through the Policy Engine.

---

### 11.1 Transaction Generation

**REQ-AEE-01:** The system shall generate transaction payloads for the following action types:

| Action | Protocol |
|---|---|
| Lending (supply) | Folks Finance |
| Borrowing | Folks Finance |
| Repayment | Folks Finance |
| Withdrawal from lending | Folks Finance |
| Token Swap | Tinyman, Pact |
| LP Provision | Tinyman, Pact |
| LP Withdrawal | Tinyman, Pact |

**REQ-AEE-02:** All transaction payloads shall be human-readable before signing.

---

### 11.2 Execution Plan Generation

**REQ-AEE-03:** Before execution, the system shall generate a human-readable Execution Plan (POA) summarizing all actions, expected outcomes, estimated costs, and risks.  
**REQ-AEE-04:** The Execution Plan shall be presented to the user for review and approval before any transaction is submitted.  
**REQ-AEE-05:** The system shall support multi-step execution plans (e.g., swap → supply → LP provision in sequence).

**Output:** Execution Plan (POA)

---

### 11.3 Transaction Simulation

**REQ-AEE-06:** Before submission, the system shall simulate each transaction to verify expected outcome.  
**REQ-AEE-07:** Simulation outputs shall include estimated results, estimated fees, and any warnings.  
**REQ-AEE-08:** If simulation reveals unexpected results, the system shall flag the discrepancy to the user before proceeding.

**Output:** Simulation Results + Estimated Fees

---

### 11.4 Transaction Execution

**REQ-AEE-09:** The system shall execute transactions only after receiving explicit user approval via the Policy Engine authorization flow.  
**REQ-AEE-10:** The system shall handle transaction failures gracefully and provide clear error messages.  
**REQ-AEE-11:** The system shall maintain an auditable transaction log for all executed actions.  
**REQ-AEE-12 (Phase 3):** The system shall support Autopilot mode for pre-authorized recurring actions, subject to Policy Engine limits.

**Output:** Transaction Result + Audit Log Entry

---

## 12. Orchestration & Execution Layer

**Purpose:** Coordinate all approved execution across protocols through a structured pipeline of policy, planning, and routing.

---

### 12.1 x402 Payment Layer

**REQ-ORC-01:** The system shall implement x402-based payment checks on all paid API endpoints.  
**REQ-ORC-02:** x402 payments shall be settled and tracked via the GoPlausible Facilitator.  
**REQ-ORC-03:** The payment check shall be resolved before the engine executes the request.

---

### 12.2 Policy Engine

**REQ-ORC-04:** All execution-intent requests shall be routed through the Policy Engine before proceeding.  
**REQ-ORC-05:** The Policy Engine shall enforce the following checks:

| Check | Description |
|---|---|
| User Approval | Confirms explicit user authorization exists |
| Transaction Policies | Validates action against user-defined transaction rules |
| Risk Limits | Rejects actions that would breach the user's configured risk limits |
| Execution Limits | Enforces per-session and daily execution caps |
| KYC Gate | Validates KYC status if execution features require it |

**REQ-ORC-06:** Any request that fails a Policy Engine check shall be rejected with a clear, user-facing reason.  
**REQ-ORC-07:** The Policy Engine shall create guardrails that prevent unauthorized or out-of-policy execution under all circumstances.

---

### 12.3 Orchestrator / Planner

**REQ-ORC-08:** Upon Policy Engine approval, the Orchestrator/Planner shall determine the Plan of Action (POA).  
**REQ-ORC-09:** The POA shall define the ordered sequence of protocol interactions required to fulfill the user's intent.  
**REQ-ORC-10:** The Orchestrator shall optimize execution order to minimize cost and slippage.

---

### 12.4 Execution Coordinator

**REQ-ORC-11:** The Execution Coordinator shall manage the execution of multi-step, multi-protocol action sequences defined in the POA.  
**REQ-ORC-12:** The Coordinator shall handle step dependencies — a subsequent step shall only execute after the preceding step succeeds.  
**REQ-ORC-13:** The Coordinator shall implement failure recovery — on step failure, the system shall halt remaining steps and surface the failure with current state to the user.

---

### 12.5 Haystack Router

**REQ-ORC-14:** The Haystack Router shall route each execution action to the correct protocol adapter.  
**REQ-ORC-15:** The Router shall support the following protocol adapters at launch: Folks Finance, Tinyman, Pact.  
**REQ-ORC-16:** Adding a new protocol adapter shall not require changes to any engine or orchestration component outside the Router.

---

### 12.6 Gora Oracle

**REQ-ORC-17:** The system shall integrate Gora Oracle to provide verified price feeds to the execution pipeline.  
**REQ-ORC-18:** All price data used during execution (slippage calculation, liquidation threshold monitoring) shall be sourced from Gora Oracle.  
**REQ-ORC-19:** Oracle failures shall result in execution being paused, not proceeding with stale or unverified price data.

## 13. AI Copilot Layer

**Purpose:** Provide a natural language interface over all six engines and the orchestration layer.

---

### 13.1 Query Routing

**REQ-COP-01:** The system shall accept natural language queries from the user.  
**REQ-COP-02:** The system shall route each query to the appropriate engine(s) based on intent classification.

**Sample query routing:**

| Query | Routes To |
|---|---|
| "What is my allocation?" | Engine 1 (Portfolio) |
| "Why is my portfolio risky?" | Engine 2 (Risk) |
| "How can I improve returns?" | Engine 3 (Strategy) |
| "Find safer yield opportunities" | Engine 4 (Yield) |
| "What's my risk profile?" | Engine 5 (User Intelligence) |
| "Rebalance my portfolio" | Engine 3 → Engine 6 → Orchestration |
| "Move idle USDC to best opportunity" | Engine 4 → Engine 6 → Orchestration |
| "Am I close to liquidation?" | Engine 2 (Risk) |

---

### 13.2 Response Requirements

Every Copilot response shall satisfy:

**REQ-COP-03:** A direct, accurate answer to the user's query  
**REQ-COP-04:** The data and reasoning behind the answer  
**REQ-COP-05:** A confidence level indicator (where the answer involves estimation or prediction)  
**REQ-COP-06:** All assumptions made in producing the answer  
**REQ-COP-07:** An optional suggested next action (e.g., "Would you like me to move the idle USDC?")

---

### 13.3 Conversation

**REQ-COP-08:** The system shall support multi-turn conversations with context retention across turns.  
**REQ-COP-09:** The system shall maintain conversation history per session.

---

### 13.4 AI Constraint Enforcement

**REQ-COP-10:** The Copilot shall never recommend or initiate execution without routing through the Policy Engine and obtaining user approval.  
**REQ-COP-11:** The Copilot shall never present a recommendation without a rationale, confidence level, and associated risks.  
**REQ-COP-12:** The Copilot shall never hide uncertainty — low-confidence outputs must be flagged as such.

## 17. Security Requirements

**REQ-SEC-01:** The system shall be non-custodial — private keys shall never be stored, transmitted, or accessed by CrestFlow systems.  
**REQ-SEC-02:** All transaction signing shall occur client-side or via Turnkey's non-custodial signing infrastructure.  
**REQ-SEC-03:** Explicit user approval shall be required before any transaction is submitted to the Algorand network.  
**REQ-SEC-04:** All sensitive user data shall be encrypted at rest using AES-256 or equivalent.  
**REQ-SEC-05:** All data in transit shall be encrypted using TLS 1.2 or higher.  
**REQ-SEC-06:** All API endpoints shall implement authentication (JWT or equivalent).  
**REQ-SEC-07:** All APIs shall implement rate limiting per user and per API key to prevent abuse.  
**REQ-SEC-08:** All executed actions shall be logged in an auditable, tamper-evident transaction log.  
**REQ-SEC-09:** The Policy Engine shall act as a mandatory guardrail — no execution action shall bypass it.  
**REQ-SEC-10:** KYC verification status shall be enforced by the Policy Engine for execution-gated features.

## 19. Scalability Requirements

**REQ-SCA-01:** Each engine shall be independently horizontally scalable without impacting other engines.  
**REQ-SCA-02:** The system shall support concurrent multi-user operation without degradation.  
**REQ-SCA-03:** Adding a new protocol integration shall require only a new protocol adapter in the Haystack Router — no changes to engine internals.  
**REQ-SCA-04:** The system shall support event-driven processing for portfolio state updates and alert generation.  
**REQ-SCA-05:** The Financial Knowledge Layer data pipeline shall be independently scalable from engine compute.  
**REQ-SCA-06 (Future):** The architecture shall support multi-chain expansion without requiring fundamental redesign.

---

## 20. Reliability Requirements

**REQ-REL-01:** If a single engine fails, all other engines shall continue operating normally.  
**REQ-REL-02:** If an external data source is unavailable (e.g., CoinGecko), the system shall degrade gracefully — displaying last known data with a staleness indicator rather than failing.  
**REQ-REL-03:** If a protocol API is unavailable, positions from that protocol shall be excluded from the portfolio snapshot with a clear indicator, not silently omitted.  
**REQ-REL-04:** All external API calls shall implement retry logic with exponential backoff.  
**REQ-REL-05:** Portfolio snapshot state shall be preserved — a failed refresh shall not overwrite or corrupt the last valid snapshot.  
**REQ-REL-06:** If Gora Oracle is unavailable during an execution attempt, the system shall halt execution and notify the user rather than proceeding with unverified price data.
